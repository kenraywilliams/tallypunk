- **GBL-08** · `MVP` · `[Agreed]` — **Human-friendly sequential display IDs (per entity).** Every **Stakeholder, Company, Pool, and Grant** carries a stable, sequential, human-readable **ID** shown as its own column and on its detail view — separate from the internal database key — so two same-named records are never confused.
  - **Zero-padded minimum widths per entity:** Company **`01`** (2 digits) · Pool **`001`** (3) · Stakeholder **`000001`** (6) · Grant **`0000001`** (7). Numbering starts at 1.
  - **No maximum — the padding is a minimum display width, not a cap.** When the count outgrows the padded range the ID simply gains a digit: e.g. companies run `…, 98, 99, 100, 101, …` while the first ninety-nine keep their padded form (`01, 02, …`); likewise pools `999 → 1000`, grants `9999999 → 10000000`, etc.
  - IDs are **monotonic per entity** (numbers are not reused after a delete). The stored value is the integer sequence; zero-padding is display-only formatting.
  - _Status:_ Stakeholder IDs are **live in the sandbox** (`000001`, shown as a column + on the detail header). Company / Pool / Grant IDs to follow as each entity is built out.

- **GBL-09** · `MVP` · `[Built]` (sandbox, 10 Jul 2026) — **Delete (hard / irreversible for now).** Distinct from **Terminate/Pause** (GRANT‑16/17 = lifecycle states, not removal). No undo yet (Trash = GBL‑03, future). Every delete **logs**; because a deleted object's own History dies with it, deletes must also write to a **global / company-level audit log** (GBL‑02) so the record survives. **Cascade rules per object:**
  - **Grant** — deletes outright and **releases its reserved units back to its pool**. Simple confirm; warn if it had vesting.
  - **Stakeholder** — **cascade-deletes their grants** (each releasing pool units), with a **scrollable summary** (grant count + units freed per pool). _Recommend a **typed "delete" confirm** since it destroys grants — flag for Mate._
  - **Pool** — grants are **moved, not killed**: default to the **General Pool** (POOL‑09), or let the user **transfer them to another existing pool** (cleaner than orphaning). Confirm by **typing "delete" + OK**, then a **scrollable summary** listing affected grants **grouped by stakeholder** and where they landed. Transfer to a **specific** pool runs a **capacity check** (over-capacity handled like GRANT‑07); the **General Pool never blocks** (infinite).
  - **Company** — **detaches, does not delete children**: clears the company field on its **pools, stakeholders, and grants**. **Simple "are you sure → OK"** (no typing) + a **summary of stakeholders & pools now with empty company**.
  _Build notes from the pre-build web analysis (10 Jul 2026):_ (1) **Prerequisite — persistent per-entity seq counters** (`nextSeq` in the sandbox JSON): seq is currently `max(existing)+1`, so deleting the highest-seq object would REUSE its ID and break GBL-08 monotonicity. (2) Sandbox **grants carry no company field** (inherited transitively) — company delete touches pools + stakeholders only. (3) Grant DELETE log summaries must **embed the display ID** ("Grant #0000019 deleted — N units returned to {pool}") since the label lookup dies with the grant; entries stay attributed on the owner's timeline. (4) Deleting a stakeholder while on their page **redirects to the list**; cascade closes dialogs. (5) General Pool enforced via an **`isGeneral` flag** (not a magic name), name reserved in the dup-check, seeded automatically, recreated on reset; pool-transfer capacity checks use **reserved** (termination-aware) units, and terminated grants moved to the General Pool are always reinstatable (infinite = pre-flight never blocks). (6) Deletes write their summary entry to the **parent's timeline** where one exists (grant→stakeholder is automatic via attribution); a global "deleted objects" view waits for GBL-02's rich audit.

- **GRANT-06** · `MVP` · `[Built]` — **Vesting calculator must total exactly 100%.** Live running total; **can't save if over/under 100**. **Model (rebuilt Jul 2026):** the **vesting period = the number of schedule rows (Year 1..N)** — every year is a real vesting year (no "dead" cliff year on top). Percentages may differ per year; frequency **subdivides** each year into steps. **Vested-as-of-today** = Σ tranche % with date ≤ today, × quantity — a true **step function** (used app-wide: totals, list "vested", chart).

- **GRANT-16** · `P1` · `[Built]` (sandbox, 3 Jul 2026) — **Terminate vesting** (part of how the system works — **distinct from Delete grant**, which is GBL‑03 later). A reversible `terminationDate` on the grant (**date required; may be today**). From that date vesting **stops**: everything vested **strictly before** the date stays vested; a tranche landing **exactly on** the termination date does **not** vest (decision locked — legally the safer default; revisit as a plan setting only if needed). The remaining unvested is **forfeited** (pool-return handled with GRANT-14/POOL work, not here). **Un-terminate** exists and pops a **warning**: "all previously-scheduled vesting will resume — use **Pause vesting** if you want a temporary hold." - **GRANT-17** · `P1` · `[Built]` (sandbox, 3 Jul 2026) — **Pause vesting** (also core, not delete). `pauseStart` (**required**) + **optional** `pauseEnd`. Vesting **freezes** from the start. **With an end date → shift the tail** (decision locked): every tranche on/after the pause is **delayed by the pause length**, so the grant **still reaches 100%, just later** (fully-vested date pushes out by the pause duration) — standard leave-of-absence treatment. **Without an end date** it looks like a Terminate (flat from the start) **but stays resumable** — add an end date later and the schedule recomputes. Engine: same **tranche-transform** approach (re-date tranches ≥ pauseStart by +duration; open-ended → suspend). **UI:** a **"Pause vesting"** button; edit/resume/remove the pause; **"Paused {start}[ → {end}]"** banner + status badge in list/detail/stakeholder views. **Chart:** a **shaded pause band** between start and end with start/end icons; curve flat across the band, then resumes shifted. **Edge cases:** `pauseEnd` before `pauseStart` invalid; pause window entirely after fully-vested → no effect; pause spanning the cliff delays the cliff boom too.
- **GRANT-18** · `P1` · `[Built]` (sandbox, 8 Jul 2026) — **Stakeholder-level Terminate / Pause vesting** (rolls up GRANT‑16/17). One action on a **stakeholder** fans the same event out to **all their grants**, tagged **"inherited"** so an un-terminate / un-pause at the person level reverses them together. **Precedence:** a grant **terminated/paused individually** wins over the inherited person-level event (don't revive or override it). All events stay **auditable** (GBL‑02 logging). This is the common real-world path (someone leaves → all their grants stop at once). **Un-terminate wizard (capacity-aware):** a grant terminated **at the grant level never auto-un-terminates** when the person is un-terminated — but it is **surfaced** so the user can opt in. Un-terminate runs a **pre-flight**: for each affected grant, can its pool re-supply the units it returned? ## J · Backups

- **BKP-01** · platform `MVP` / user-facing `P2` · `[Agreed]` — **Three things, kept separate:**
  1. **Platform backups protect EVERYONE** (free + paid): nightly DB snapshots + an automatic snapshot **before every import**. Our disaster recovery — "if we screw up, even free users don't suffer." Controlled restore window ~**4 weeks** (cost-bound).
  2. **User self-export (download) anytime, unlimited** (CSV/JSON, their PC, their dime) — this is how a paid user keeps "more than 4 weeks" without us storing it. Cheap for us (generated on demand).

So "restore my export" = re-import with validation + dry-run + atomic write (IMP-01/02). Our controlled snapshot-restore stays for true disasters only.
